# IDEA_docs_v2_refresh — Docs v2 Refresh (PRD v6 Aligned)

**Owner:** Product + Engineering  
**Created:** 2026-02-27  
**Status:** Draft (Idea captured)  
**Primary reference:** `docs/PRD_v6.0.md` (canonical PRD v6)  

---

## 1. Problem & Motivation

The current `docs/` and legacy `docs/plan/**` tree grew organically over early MVP phases. As a result:

- Important information is **duplicated or split** across multiple PRDs and specs.
- Some documents are **out of date** relative to `PRD_v6.0.md`.
- The planning folder mixes **historical artifacts** (phase-1 MVP, resident portal v1, marketing experiments) with current work.
- New contributors struggle to find:
  - “What is the current truth?” (PRD, security rules, flows).
  - “What is the current plan?” (phases, prompts, roadmaps).

We need a **Docs v2** set that is:

- **Aligned to PRD v6.0** (single source of product truth).
- **Minimal but complete** (no redundant or stale docs).
- **Plan-aware** (plays nicely with `/idea`, `/plan`, `/dev`, `/ship` flows).

---

## 2. Goals

1. **Single canonical product spec**
   - `docs/PRD_v6.0.md` is the only PRD for GateFlow core (plus archived historical PRDs).

2. **Clean, navigable docs tree**
   - `docs/` contains a small set of high-signal guides:
     - Architecture, Security, Development Guide, Environment Variables.
   - Legacy content lives under `docs/archive/**`, not mixed with v2 docs.

3. **Plan-ready `docs/plan/`**
   - New `docs/plan/` structure is empty and ready to be populated exclusively via:
     - `/idea` → context (IDEA_*.md)
     - `/plan` → plans (PLAN_*.md)
     - `/dev` → phase prompts (PROMPT_*_phase_*.md)
   - Old plan docs remain accessible under `docs/archive/plan-legacy/**`.

4. **Security & gated-community focus**
   - Docs v2 highlights why GateFlow is a **must-have security tool** for gated communities:
     - Visitor identity & trust.
     - Guard accountability & incidents.
     - Scanner/gate policies & hardware integration.
     - Resident safety and privacy.

---

## 3. Scope (Docs v2)

### 3.1 In scope

- **Docs root cleanup**
  - Keep: `docs/PRD_v6.0.md`.
  - Create: `docs/guides/` with a small, curated set:
    - `ARCHITECTURE.md`
    - `SECURITY_OVERVIEW.md`
    - `DEVELOPMENT_GUIDE.md`
    - `ENVIRONMENT_VARIABLES.md`
  - Move all other v1/v5-era docs into `docs/archive/**` (already started with `plan-legacy`).

- **New plan structure**
  - `docs/plan/context/IDEA_<slug>.md` — initiative context (this file is IDEA_docs_v2_refresh).
  - `docs/plan/execution/PLAN_<slug>.md` — per-initiative plan.
  - `docs/plan/execution/PROMPT_<slug>_phase_<N>.md` — per-phase prompts, using `TEMPLATE_PROMPT_phase.md`.
  - `docs/plan/learning/{patterns,incidents,decisions}.md` — cross-phase learning docs (rebuilt/cleaned later).

- **Cursor flows alignment**
  - Ensure `/idea`, `/plan`, `/dev`, `/ship` flows:
    - Read `PRD_v6.0.md` and `docs/plan/context/*` as canonical product + initiative context.
    - Write to `docs/plan/execution/*` using existing templates and new skills (e.g. `pro-prd-writer`).

### 3.2 Out of scope (Docs v2 idea only)

- Implementing code changes described by PRD v6 (e.g., scanner rules, resident mobile).
- Writing every possible guide up front; we start with the **core set** and expand incrementally.

## 5. Success Criteria

- **Findability**
  - A new engineer can answer:
    - “What is GateFlow?” by reading `PRD_v6.0.md` and `ARCHITECTURE.md`.
    - “How do I run/dev/test?” by reading `DEVELOPMENT_GUIDE.md`.
    - “What are the security invariants?” by reading `SECURITY_OVERVIEW.md` + `.cursor/rules/*`.

- **Clarity**
  - No conflicting PRDs (v5 vs v6); v5 lives under archive and is clearly “historical”.
  - “Docs v2” structure is obvious from `docs/README.md` (to be generated) and matches PRD v6.

- **Maintainability**
  - New initiatives always start via `/idea` and `/plan` and land in `docs/plan/**` using shared templates.
  - PRD changes are done via `pro-prd-writer` skill and kept in sync with guides.
